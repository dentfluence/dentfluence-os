# Phase 3 — Task Engine Human/System split

Blueprint Phase 3 ("Work surfaces") has three deliverables. Two were built as part of Workstream E: the Today's Actions projection (E1–E5, live) and the Daily Huddle rewire (E4, live). This closes the third: the Task Engine gains a Human/System classification, behind the `tasks.human_system_split` flag (declared in `config/features.php`, default off).

## What "Human" vs "System" means

- **Human** — a person must act on this. Default for every task: manual (TaskController::store), Practice Protocol, Lab, PO/Inventory, TreatmentVisit, AppointmentReminderEngine, and the Tulip assistant's `create_task` tool. All of these are auto-generated by *something*, but a staff member still has to do the actual work (service the AC, call the patient, etc.), so they stay visible.
- **System** — a record of something Automation (RulesEngine / the Automation Engine, via `ReminderEngine`) already did. Only tasks created through `TaskEngine::autoCreate()` get this tag. Hidden from reception-facing "my work" lists once the flag is on.

This is a narrower rule than "anything auto-generated is System" — it matches the blueprint's exact wording ("System Tasks are records executed by Automation") and means only one integration point needed tagging.

## What was built

| Piece | File |
|---|---|
| Migration | `database/migrations/2026_07_03_100001_add_task_type_to_tasks_table.php` — adds `task_type` (`human`\|`system`, default `human`) + index; backfills existing tasks tagged `system` using the `task.auto_created` / `reminder.created` Activity log trail (no guessing). |
| Model | `app/Models/Task.php` — `TASK_TYPES` const, `scopeHuman()`, `scopeSystem()`, `scopeVisibleToReception()` (flag-aware — off = unchanged, on = human only), `isSystemTask()` / `isHumanTask()`. |
| Tagging | `app/Services/Relationship/TaskEngine.php::autoCreate()` sets `task_type = 'system'`. `ReminderEngine` delegates to `autoCreate()`, so recall/membership/appointment-confirm/birthday/lab-followup reminders inherit the tag automatically. |
| Reception surfaces gated | `TaskController::index` (`/tasks`), `TaskController::myTasks` (`/tasks/my`), `ListTasksTool` (Tulip assistant `list_tasks`), `HuddleService::tasksSection()` (Daily Huddle "Tasks & Callbacks" report) — all call `->visibleToReception()`. |
| Tests | `tests/Feature/Relationship/TaskHumanSystemSplitTest.php` — tags correctly set on create; reception list shows System tasks with flag off, hides them with flag on; My Tasks page same; `scopeVisibleToReception` unit-level check. |

**Deliberately NOT touched:** `AppointmentReminderEngine::generateReminders()` creates "Reminder call" tasks via `Task::create()` directly (not through `TaskEngine::autoCreate()`) — these stay `human`, because someone still has to make the call. The Huddle Task board (`HuddleTaskLog`) only ever contains tasks created via `TaskController::store`/`uploadEvidence`, so `TaskEngine`-created tasks never reach it today regardless of this change.

## Verify (you)

```
php artisan migrate
php artisan test --filter=TaskHumanSystemSplitTest
```

Then, optionally, flip the flag and confirm on real data:

```php
\App\Support\Features\Feature::set('tasks.human_system_split', true);
```

Visit `/tasks` and `/tasks/my` — Automation-created (System) tasks should disappear from both lists. Flip back off (`Feature::set('tasks.human_system_split', false)`) to confirm the rollback is instant and lossless (no data is deleted, only the read filter changes).

**Phase 3 (Work surfaces) is now feature-complete**: Today's Actions projection (Workstream E) + Daily Huddle rewire (Workstream E) + Task Engine Human/System split (this doc). Flag still off by default — same soak-then-flip pattern as every other Phase 1–3 cutover.
